# TrendRadar 数据桥接设计

**日期：** 2026-07-21  
**状态：** 已确认，待实施

## 目标

在不申请第三方新闻 API、也不增加新的 GitHub Secret 的前提下，让现有“每日新闻推送系统”在 GitHub Actions 中使用 TrendRadar 的开源抓取能力，补充微博热搜、华尔街见闻与财联社数据。

现有系统继续负责 DeepSeek 筛选和中文整理、静态网页、企业微信 Markdown 推送与 GitHub Pages 部署。TrendRadar 不独立推送消息、不生成独立网页，也不接管定时任务。

## 方案

### 1. Actions 中临时获取固定版本源码

工作流在运行日报前，检出一个固定提交的 TrendRadar 源码至临时目录。固定版本避免上游更新突然改变内部接口导致日报中断；后续升级由本仓库显式修改版本号完成。

### 2. 数据桥接脚本

新增桥接脚本，只读取 TrendRadar 的以下公开抓取能力：

- 微博热搜：供“热搜汇总”板块使用，最多保留 5 条。
- 华尔街见闻：归入“金融财经”候选新闻。
- 财联社：归入“金融财经”候选新闻。

桥接脚本将结果标准化为本项目的 JSON 格式，包含标题、来源、链接、简短正文或简介、抓取时间与来源平台。它不调用 TrendRadar 的 AI、消息推送、存储或网页生成模块。

### 3. 现有日报的接入方式

`main.py` 读取桥接 JSON：

- 华尔街见闻、财联社项目与 RSS 项目合并，作为财经候选交给现有 DeepSeek 筛选与内容生成流程。
- 微博热搜优先取桥接结果；桥接结果缺失时，再尝试现有热搜来源；两者都失败则在日报中明确显示“今日未获取到热搜”。
- 保留 RSS、网页、企业微信、DeepSeek 与 GitHub Pages 现有流程。

## 可靠性与回退

- 每个平台单独捕获失败，不能因一个站点反爬、超时或上游结构变化而终止日报。
- 桥接脚本即使完全失败，也必须写出结构正确的空 JSON；主程序继续生成日报。
- 日志记录：TrendRadar 版本、每个平台数量、失败平台与原因、实际进入 DeepSeek 候选池的数量。
- 页面和企业微信不得由 TrendRadar 直接发送，以免重复推送。

## 许可证与部署边界

TrendRadar 使用 GPL-3.0 许可证。本项目只在运行时检出并执行其原始源码，不将其源码复制进本仓库，也不修改其代码。若未来需要修改或打包其源码，应单独遵守其许可证要求。

本阶段适用于 GitHub Actions 试跑。由于国内站点可能限制 GitHub 的海外出口 IP，失败属于可预期的外部情况，并会触发上述回退。迁移到 NAS 后，日报脚本与 TrendRadar 可在同一 Docker 环境中运行，数据接口和 JSON 格式不变。

## 验收标准

手动运行一次 GitHub Actions 后：

1. 日志显示 TrendRadar 固定版本和微博、华尔街见闻、财联社的逐项抓取数量。
2. 即使任一平台失败，日报、网页部署和企业微信推送仍正常完成。
3. 网页与企业微信热搜中优先显示微博热搜；财经候选的来源中可出现华尔街见闻或财联社。
4. 不新增 GitHub Secrets，不出现 TrendRadar 的重复推送或额外网页。
